# P2P Protocols Skill

You are an expert in peer-to-peer network architecture and protocol implementation.
Apply rigorous thinking about adversarial conditions, network partitions, and churn.

## Core Conceptual Stack

### Peer Identity
- Peer ID generation and verification (public key based preferred)
- Identity persistence vs. ephemeral sessions
- Sybil resistance considerations
- **Project:** PeerJS auto-generates a UUID per `new Peer()` call. IDs are ephemeral (new ID on every page load/session). No identity persistence, no public-key binding. The host's peerId is the "room code" — shared manually or via `?join=PEER_ID` URL param. No Sybil resistance needed (2-peer direct games, no routing table).

### Discovery Layer
Choose and document the approach:
- **Bootstrap nodes** — static entry points, single point of failure risk
- **mDNS / LAN discovery** — local network only
- **DHT (Kademlia-style)** — scalable, decentralised
- **Rendezvous servers** — centralised coordination, decentralised data
- **PEX (Peer Exchange)** — gossip-based, requires initial contact
- **Project:** Rendezvous via PeerJS free signaling server (`0.peerjs.com`). Host gets a UUID peerId (auto-generated by PeerJS, standard UUID v4 format), shares it as a room code and/or `?join=PEER_ID` URL. Guest enters the code to call `peer.connect(hostId)`. Signaling is used only for WebRTC handshake (~1 KB); all game data flows over WebRTC DataChannel directly.

### Transport & Connectivity
- Protocol preference: TCP / QUIC / WebRTC / WebSocket
- **NAT traversal strategy:**
  - STUN for address discovery
  - TURN as relay fallback
  - ICE for candidate negotiation
  - Hole punching (TCP simultaneous open, UDP hole punching)
- TLS / Noise protocol for encryption
- **Project:** WebRTC DataChannel via PeerJS v1.5.5. Encrypted by default (DTLS-SRTP). ICE candidate negotiation handled by PeerJS. STUN/TURN from PeerJS default ICE config. No custom TURN server configured — relies on PeerJS defaults (may fail behind symmetric NAT).

### Routing & Overlay
- **DHT routing:** Kademlia XOR metric, k-buckets, iterative vs. recursive lookup
- **Flooding / gossip:** fanout, TTL, deduplication (seen-set or bloom filter)
- **Structured overlay:** Chord, Pastry, Tapestry tradeoffs
- **Project:** No overlay routing — topology is a single direct WebRTC DataChannel between exactly 2 peers (host + guest). No DHT, no gossip, no relay beyond WebRTC ICE.

### Message Protocol
Design checklist:
- [ ] Message framing (length-prefixed / delimiter / QUIC streams)
- [ ] Versioning and backward compatibility
- [ ] Request-response correlation IDs
- [ ] Idempotency keys for retries
- [ ] Max message size and chunking strategy
- [ ] Serialisation: protobuf / msgpack / JSON / custom binary
- **Project serialisation:** Two message types over the same DataChannel:
  1. **Game moves** — plain UCI strings: `"e2e4"` (normal), `"e7e8q"` (promotion). No framing, no ID. Encoded/decoded by `uciUtil.ts`.
  2. **Protocol messages** — JSON strings: `{"type":"init","game":"chess"}` and `{"type":"disconnect","role":"host"|"guest"}`. Parsed in `P2PContext.tsx` `handleMessage`. Anything that fails JSON.parse is treated as a game move.

### Pubsub / Gossip
If using pubsub (e.g. GossipSub):
- Topic subscription and unsubscription lifecycle
- Message validation before forwarding
- Heartbeat and IHAVE/IWANT message exchange
- Score-based peer selection (penalise slow / malicious peers)
- **Project:** No pubsub. All communication is direct point-to-point over a single DataChannel. No topics, no fanout, no gossip.

## Churn & Fault Tolerance
- Assume peers drop without notice — use heartbeats with short timeout
- Replicate data to `k` neighbours (k=20 in standard Kademlia)
- Re-announce data periodically before expiry
- Detect and quarantine slow / unresponsive peers
- **Project:** 2-peer game — any peer dropping = game over. `conn.on("close")` in `useP2PGame.ts` drops status to `"waiting"`, triggering "Connection lost." modal in UI. No reconnection logic currently. Intentional leave sends `{"type":"disconnect","role":...}` JSON before closing.

## Security Considerations
- Eclipse attacks — diversify routing table (multiple /24 subnets)
- Sybil attacks — proof-of-work, proof-of-stake, or identity binding
- Routing table poisoning — validate XOR distances
- Message amplification — rate limit by peer
- Resource exhaustion — cap connections, cap routing table size
- **Project threat model:** Low-stakes public board game. No financial data, no auth. Primary risks: (1) Guest impersonation — not possible to prevent; by design peerId is shared voluntarily; (2) Malformed UCI strings causing `decodeMove` to return garbage — engine guards against bad moves via `moveSet.find()` returning undefined; (3) PeerJS signaling server outage — game cannot start (no fallback); (4) Symmetric NAT failure — no TURN fallback configured.

## Observability
Key metrics to instrument:
- Connected peer count (target range: exactly 2 — host + guest; any other value = error state)
- ~~DHT lookup latency~~ — not applicable (no DHT)
- Move delivery latency (WebRTC DataChannel — typically <100ms LAN, <300ms WAN)
- Peer churn rate — binary per game session (connection alive or dead)
- NAT traversal success rate — unmonitored currently; PeerJS default ICE config
- **Project:** No observability infrastructure. All state is in React (`P2PStatus`: `"idle" | "waiting" | "connected" | "error"`). Add browser console logging for debugging connection issues.

## Implementation Reference
- **libp2p:** modular, multi-language, production-proven
- **go-libp2p / rust-libp2p / js-libp2p** — language-specific
- **WebRTC** — browser-compatible, STUN/TURN required
- **Project:** PeerJS v1.5.5 (`peerjs` npm package). Wraps WebRTC DataChannel. Dynamically imported in `useP2PGame.ts` to avoid SSR issues (`import("peerjs")`). No server-side component.
